## 002｜签名、授权、交易

这一节的核心是区分 `Signature`、`Approval` 和 `Transaction`，不要把钱包里的“签一下”理解成普通 Web2 登录。

### Signature

`Signature` 是用私钥对某段具体内容进行签名，用来证明“我控制这个地址，并确认了这段内容”。签名通常不上链、不消耗 gas，但它不是天然无风险的登录按钮。恶意签名可能变成订单、授权声明、Permit 或其他可被利用的链下承诺，因此签名前必须检查消息内容、请求域名、请求方、过期时间和用途。

### Approval

`Approval` 通常是授权某个合约或 spender 使用某种 token 的额度，例如允许一个合约最多转走多少 USDC。它不是一次性付款，而是给出资产使用权限。常见 ERC-20 `approve` 是链上交易，会消耗 gas，并且可能长期有效；如果授权额度过大或 spender 合约有风险，资产可能被持续转走。

### Transaction

`Transaction` 是真正发送到链上执行的操作，例如转账、调用合约、写入状态或修改授权。它通常会上链、消耗 gas，并且成功后会改变链上状态；即使失败，也可能已经消耗 gas。因此交易必须由用户人工确认，不能交给 AI Agent 自动执行。

## 对比表

| 动作类型 | 是否上链 | 是否消耗 gas | 是否影响资产或权限 | 是否必须人工确认 |
|---|---|---|---|---|
| Signature 签名 | 通常否 | 通常否 | 可能是 | 是 |
| Approval 授权 | 是 | 是 | 是 | 是 |
| Transaction 交易 | 是 | 是 | 是 | 是 |
| 查询余额 | 否 | 否 | 否 | 否 |
| AI 生成交易参数 | 否 | 否 | 否 | 否 |

## AI x Web3 安全边界

AI Agent 可以帮助分析风险、解释合约参数、生成交易草稿或提醒用户检查关键字段，但不能直接替用户签名、授权、转账或写入合约。AI 生成交易参数本身不危险，风险发生在“拿这些参数去签名、授权或发交易”的那一步。

涉及钱包弹窗时，至少检查：

- 当前网络是否正确。
- 接收地址或合约地址是否可信。
- 资产类型和数量是否正确。
- Approval 的 spender 和额度是否合理。
- 交易函数名、参数和权限影响是否能理解。
- 是否存在无限授权或长期有效权限。

## Hackathon 风险提示

任何 Agent Wallet 或 AI Payment 设计都必须保留 human-in-the-loop。可接受的默认流程是：

```text
AI 分析任务 -> AI 生成交易草稿 -> 用户复核参数 -> 钱包弹窗 -> 用户人工确认 -> 链上执行 -> 记录交易 hash
```

不应作为默认方案的流程是：

```text
AI 持有私钥 -> AI 自动签名 -> AI 自动授权 -> AI 自动转账
```
