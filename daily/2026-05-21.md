## 003｜Gas、网络、测试网、区块浏览器

这一节的核心是理解链上执行不是普通后端 API 调用。一次链上操作通常要经过：用户发起、钱包确认、选择网络、支付 Gas、网络打包执行、生成 transaction hash，并最终通过区块浏览器验证状态。

## 核心概念

### Gas

`Gas fee` 是链上操作时支付给网络验证者 / 节点的执行资源费用。它和转账金额不是一回事：转账金额是给接收方的资产，Gas 是让网络执行这笔交易所需要付出的成本。即使交易失败，也可能因为网络已经尝试执行而消耗 Gas。

### Testnet

`Testnet` 是用于练习操作、测试程序和验证流程的测试网络。测试网里的代币通常没有真实经济价值，但测试网不是用来养成随意点击习惯的地方。测试时也应该检查网络、地址、金额、合约和授权信息，因为这些习惯会迁移到主网。

### Transaction Hash

`Transaction Hash` 是一笔交易提交或广播后生成的查询标识，可以用于在区块浏览器中追踪交易。需要注意：有 transaction hash 不等于交易已经成功。最终仍然要查看区块浏览器中的状态，例如 `Success`、`Failed`、确认数、区块高度、Gas 使用量和调用信息。

### Block Explorer

`Block Explorer` 是区块浏览器，用来查询链上公开数据。拿 transaction hash 去区块浏览器，可以查看交易是否成功、由哪个地址发起、发给哪个地址或合约、消耗了多少 Gas、在哪个区块、调用了什么函数，以及是否触发事件。

## 判断题沉淀

| 说法 | 判断 | 原因 |
|---|---|---|
| 有 transaction hash 就说明交易成功了 | 不对 | hash 只是查询标识，最终要看 `Success / Failed` 等状态 |
| 测试网交易不需要检查钱包弹窗 | 不对 | 测试网也要培养主网级别的安全习惯 |
| AI Agent 可以生成交易草稿，但不能替我确认钱包 | 对 | 生成草稿和执行交易是两件事，钱包确认必须 human-in-the-loop |
| 区块浏览器可以用来验证交易是否真的执行成功 | 对 | 区块浏览器能查看状态、区块、Gas、调用对象和交易详情 |

## 链上执行流程

```text
用户发起交易
-> Agent 辅助生成交易草稿或解释参数
-> 用户检查网络、地址、金额、合约、Gas 和风险
-> 用户在钱包中人工确认
-> 交易发送到具体网络
-> 网络打包执行并记录 Gas 使用
-> 生成 transaction hash
-> 用户或 Agent 用 transaction hash 打开区块浏览器
-> 查询 Success / Failed、区块高度、Gas、调用对象和关键参数
-> 将 explorer 链接或交易信息作为 proof 返回用户
```

## AI x Web3 安全边界

AI Agent 可以做的是：

- 解释 Gas、网络、交易参数和区块浏览器字段。
- 生成交易草稿。
- 检查参数是否存在明显风险。
- 根据 transaction hash 帮用户整理验证材料。

AI Agent 不应该做的是：

- 持有私钥或助记词。
- 自动点击钱包确认。
- 在用户未复核时自动发起主网交易。
- 只记录“已完成”，但不保存 transaction hash、explorer 链接或状态信息。

## Hackathon 风险提示

AI x Web3 项目不能只展示“Agent 说执行完成”。更可靠的 proof 应包含：

- network
- transaction hash
- block explorer link
- status
- gas used
- from / to 或合约地址
- 用户人工确认点
