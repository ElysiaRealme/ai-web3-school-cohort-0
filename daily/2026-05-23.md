## 最小路径

用责任边界区分 Prompt、Workflow 和 Agent：

| 类型 | 决策权 | 执行方式 | 适合任务 | 主要风险 |
| --- | --- | --- | --- | --- |
| Prompt | 人 | 单次模型输出 | 概念解释、草稿生成、轻量转换 | 幻觉、不可复现 |
| Workflow | 人预设流程 | 固定或基本固定的步骤 | WCB 查询、笔记生成、proof 打包、重复性学习任务 | 异常处理脆弱 |
| Agent | 模型根据中间状态动态规划并调用工具 | 多步骤动态执行 | 多资料读取、工具选择、长期状态维护 | 越权执行、状态污染、错误链式放大 |

## 推荐路径

学员回答与修正：

1. WCB 任务查询更接近 Agent 任务，因为需要 Tool Use 和真实 API 调用。
   - 修正理解：只读 API 调用本身可以是 workflow 的一个步骤；但“决定查什么、解释状态、选择下一步学习行动”的完整链路更接近 agentic workflow。
2. 读取 Notion、WCB、Handbook 并规划 Week 1 路线是 Workflow，因为链路可重复，步骤由人预设。
   - 修正理解：这是一个 agent-assisted workflow。路线是固定的，但 Agent 可以负责重试、抽取、对照和总结。
3. AI 可以生成合约交互脚本，但不能自动发送交易。
   - 人工复核必须发生在钱包确认之前。
   - 复核内容包括网络、合约地址、函数名、参数、金额、授权额度、接收方和预期状态变化。
   - 如果学员没有理解自己确认的内容，单纯“钱包点确认”不等于真正复核。
4. Learning Agent 的 proof 材料至少应包括：
   - Markdown 笔记。
   - GitHub commit hash。
   - 公开 GitHub 链接。

建议额外保留的 proof 材料：

- 规划时使用的来源链接。
- WCB 只读状态快照。
- 工具调用或行动日志摘要。
- 涉及 WCB 或链上执行时的截图、交易哈希或区块浏览器链接。
- 明确记录 Agent 做了什么，以及人确认了什么。

## 挑战路径

受限 Web3 学习助手的设计规则：

允许：

- 读取 WCB 任务状态。
- 读取 Handbook、Notion 和公开资料。
- 生成笔记、清单、概念卡片和脚本草稿。
- 解释交易、ABI、合约调用和区块浏览器记录。
- 准备 GitHub proof 材料。

未经明确人工动作禁止：

- 接触私钥或助记词。
- 代签消息。
- 代授权 Token allowance。
- 代发转账。
- 部署合约或写入合约。
- 不经预览和确认就提交 WCB 写入操作。

必须人工确认的节点：

- 任何 WCB 写入操作之前。
- 任何钱包签名之前。
- 任何授权之前。
- 任何转账之前。
- 任何合约写入之前。
- 任何可能包含敏感信息的公开 proof 提交之前。

## 笔记

有用的区分不是“这个任务到底叫 Agent 还是 Workflow”，而是决策权在哪里：

- 如果人定义每一步，系统只按步骤执行，它更接近 Workflow。
- 如果模型根据中间状态选择下一步和调用哪个工具，它更接近 Agent。
- 如果任务触及 Web3 权限，自动化程度越高，就越需要安全控制、日志和人工确认点。

在本课程中，Learning Agent 可以在读取、规划、总结和 proof 打包环节保持 agentic；但在签名、授权、转账和合约写入环节必须保持人工门控。
