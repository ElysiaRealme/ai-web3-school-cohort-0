## 004｜AI Agent 到 Web3 的最小安全工作流

本节不作为新知识展开，而是把前三节合并为一个安全执行原则：

```text
AI 生成 -> 人工复核 -> 钱包确认 -> 链上执行 -> 区块浏览器验证
```

## 综合理解

AI x Web3 的关键不是让 Agent 尽可能自动化，而是让 Agent 在不接触私钥、不代替用户签名、不自动转账的前提下，提高用户理解和执行效率。

Agent 可以做：

- 解释交易参数、Gas、网络、合约地址和区块浏览器字段。
- 生成交易草稿或操作建议。
- 帮用户检查明显风险。
- 根据 transaction hash 整理 proof。

Agent 不应该做：

- 保存或读取 seed phrase / private key。
- 自动签名、自动授权、自动转账。
- 在用户未复核时发起合约写入。
- 只记录“执行成功”，但不提供 transaction hash、explorer link 和状态。

## 必须人工确认的节点

至少这些动作必须保留 human-in-the-loop：

| 动作 | 为什么必须人工确认 |
|---|---|
| 签名 Signature | 用户需要确认签名内容、域名、用途和风险 |
| 授权 Approval | 授权可能长期有效，并影响 token 使用权限 |
| 转账 / 合约写入 Transaction | 会真正上链执行、消耗 Gas，并可能改变资产或状态 |

## Hackathon 设计规则

任何 Agent Wallet、AI Payment、自动化交易或链上执行 demo，都应默认采用“Agent 准备、用户确认、链上验证”的模式：

```text
用户提出目标
-> Agent 生成计划或交易草稿
-> 用户检查网络、地址、金额、Gas、授权额度和合约函数
-> 用户在钱包里人工确认
-> 链上执行
-> 用 transaction hash 在 block explorer 验证
-> 将 explorer link / status / gas used / network 作为 proof
```
