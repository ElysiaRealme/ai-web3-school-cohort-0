## 004｜AI Agent 到 Web3 的最小安全工作流

本节不作为新知识展开，而是把前三节合并为一个安全执行原则：

```text
AI 生成 -> 人工复核 -> 钱包确认 -> 链上执行 -> 区块浏览器验证
```

## Consolidated Understanding

AI x Web3 的关键不是让 Agent 尽可能自动化，而是让 Agent 在不接触私钥、不代替用户签名、不自动转账的前提下，提高用户理解和执行效率。

Agent 可以做：

- 解释交易参数、gas、网络、合约地址和区块浏览器字段。
- 生成交易草稿或操作建议。
- 帮用户检查明显风险。
- 根据 transaction hash 整理 proof。

Agent 不应该做：

- 保存或读取 seed phrase / private key。
- 自动签名、自动授权、自动转账。
- 在用户未复核时发起合约写入。
- 只记录“执行成功”，但不提供 transaction hash、explorer link 和状态。

## Required Human Confirmation Points

至少这些动作必须保留 human-in-the-loop：

| 动作 | 为什么必须人工确认 |
|---|---|
| 签名 Signature | 用户需要确认签名内容、域名、用途和风险 |
| 授权 Approval | 授权可能长期有效，并影响 token 使用权限 |
| 转账 / 合约写入 Transaction | 会真正上链执行、消耗 gas，并可能改变资产或状态 |

## Hackathon Design Rule

任何 Agent Wallet、AI Payment、自动化交易或链上执行 demo，都应默认采用“Agent 准备、用户确认、链上验证”的模式：

```text
用户提出目标
-> Agent 生成计划或交易草稿
-> 用户检查网络、地址、金额、gas、授权额度和合约函数
-> 用户在钱包里人工确认
-> 链上执行
-> 用 transaction hash 在 block explorer 验证
-> 将 explorer link / status / gas used / network 作为 proof
```

## 005｜WCB 活动 Proof：5.22 上麦分享学习感受或信息观察

本次准备的分享主题是：使用 Codex 进行 Vibe Coding 的使用心得与技巧。核心不是把 AI 当成一次性代码生成器，而是把 Codex 当成可以分工、沉淀、复盘和持续迭代的工程协作者。

### 分享要点 1：深度链接与 Subagent 分工

一个有效的 Codex 工作流是把“规划”和“实施”拆开：

- 主 Agent 使用更强模型负责理解项目、制定北极星目标、拆解任务、验收结果。
- 实施 Agent 使用 coding 模型负责具体修改、测试、修复和报告。
- 两者通过 Codex 深度链接连接，主 Agent 向实施 Agent 下发任务，实施 Agent 完成后返回报告。

这个模式的价值是：高智能 Agent 负责方向判断和任务拆解，小模型负责具体执行，可以降低 token 和额度消耗，同时让上下文边界更清晰。

实施 Agent 的启动提示词可以保持很短：

```text
你是这个项目的实施 Agent，我会在接下来给你规划 Agent 的深度链接，它会给你发送指令让你执行。
请注意：在执行项目时，全程使用 UTF-8 编码。
```

或者更省 token：

```text
请回复我收到
```

主 Agent 的关键提示词结构是：

```text
你是这个项目的规划 Agent。
请阅读并了解这个目录下的项目。
现在制定这个项目的北极星目标：
【填写本次目标】

在检查和规划完毕后，通过 codex://threads/<implementation-agent-thread-id> 调用实施 Agent 去执行。
除非遇到必须人类手动操作的事务，否则不要停下来只汇报或询问。
不要改变实施 Agent 的模型选择，也不要采取可能会改变其模型选择的唤起方式。
```

### 分享要点 2：把历史 Codex 项目提炼成 Skill

另一个高价值用法是把历史 Codex 会话沉淀成可复用 skill。思路是按项目目录逐个扫描历史 session，以单个 session 为最小分析单位，以项目为最大归纳单位，提炼那些重复出现、可迁移、可复用的工作方法。

这个流程适合使用 goal 模式和 multi-agent，因为它需要跨 session 视野，也非常消耗 token。关键要求是：

- 每一轮只处理一个项目，保持上下文干净。
- 每个项目可以产出多个 skill，也可以产出 0 个 skill。
- skill 先按项目分类落盘，不直接安装。
- 最后再把“从 Codex session 提炼 skill”的能力本身整理成一个可复用 skill。
- 配套维护一个本地处理记录表，避免后续复用时重复扫描已经处理过的 session。

这个方法的意义是：Codex 不只是做一次任务，而是能把过去解决问题的经验变成下一次可调用的工程资产。

### 分享要点 3：学习助手 Prompt 模板

学习类任务也可以用 Codex / Hermes 形成稳定模板。一个好用的学习助手 prompt 应该明确：

- 当前学习进度。
- 学习目标和用户画像。
- 目录推进规则：不跳课、不一次讲多节、不擅自改目录。
- 教学结构：主线、原因、最小例子、常见错误、真实项目场景、练习、自测标准。
- 纠错顺序：先指出错误位置，再解释原因，再给修正方向。
- 阶段推进规则：用户说“继续”才进入下一节。

这种模板的价值是把“学习陪跑”从闲聊变成可复盘的结构化过程，尤其适合 Rust、Web3、AI Agent、后端工程等需要长期递进学习的主题。

### 分享要点 4：Hermes 与 Codex 的同步方案

我对 Hermes 和 Codex 的分工理解是：

- Hermes 更适合学习陪跑、自动复盘、学习摘要和 skill 草案。
- Codex 更适合工程实现、测试、重构、Hackathon demo 和提交前验证。
- 二者之间可以通过学习仓库里的结构化文件同步，而不是直接共享敏感上下文。

推荐的同步文件包括：

- `current-state.md`：当前状态。
- handoff 模板：跨工具交接。
- decision log：关键决策。
- verification log：验证证据。
- submission draft：提交草稿。

实施可以分 5 个阶段：

1. 建立共享上下文契约。
2. 跑通一次手动同步。
3. 增加 skill candidate pipeline。
4. 接入 Hackathon build loop。
5. 后续再考虑 MCP / API 化。

关键风险边界是：不同步密钥、助记词、WCB secret、验证码、未公开会议链接；不自动启用第三方或 AI 自动生成的 skill；WCB 提交、GitHub push、钱包签名和链上交易仍然保留 human-in-the-loop。
