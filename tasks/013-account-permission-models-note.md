# 013｜EOA、智能账户、多签：权限模型对比

日期：2026-05-26

状态：已完成。

## 本节目标

本节只解决一个问题：

```text
不同账户模型如何控制资产和权限，以及它们对 AI x Web3 助手有什么影响。
```

本节范围：

- EOA：普通外部账户。
- 智能账户：由合约控制的账户。
- 多签：需要多人或多把 key 共同批准的账户。

暂不展开完整 ERC-4337、Safe 生态或账户抽象工程实现，只理解权限模型和安全差异。

## 核心概念

### 1. EOA

EOA 是 Externally Owned Account，通常就是普通钱包账户。

特点：

- 由一把私钥控制。
- 私钥能签名，就能发交易。
- 结构简单，几乎所有钱包和 DApp 都支持。
- 风险集中：私钥丢失或泄露，账户控制权基本就丢失。

### 2. 智能账户

智能账户是由智能合约逻辑控制的账户。

特点：

- 权限规则可以写在合约里。
- 可以支持限额、白名单、Session Key、批量操作、社交恢复等。
- 更适合给 Agent 或自动化 workflow 设置边界。
- 复杂度更高，需要检查合约实现、升级权限和模块权限。

### 3. 多签

多签账户要求多个签名中的一部分共同批准操作。

例子：

```text
3 个 owner 中至少 2 个同意，交易才能执行。
```

特点：

- 单个私钥泄露不一定直接导致资产被转走。
- 适合团队金库、项目资金、DAO treasury。
- 流程更慢，需要多人确认。
- 仍需关注 owner 分布、阈值设置、模块权限和社会工程风险。

## 对比表

| 模型 | 谁控制账户 | 优点 | 主要风险 | 适合场景 |
|---|---|---|---|---|
| EOA | 单把私钥 | 简单、通用、成本低 | 私钥单点失败 | 个人测试、小额操作 |
| 智能账户 | 合约逻辑 | 可编程权限、限额、恢复 | 合约 bug、升级权限、模块风险 | Agent 钱包、自动化 workflow |
| 多签 | 多个 owner + 阈值 | 降低单点私钥风险 | 流程慢、owner 串通或管理混乱 | 团队金库、项目资金 |

## AI x Web3 连接点

AI Agent 如果直接控制 EOA，风险很高：

```text
一把 key -> 可签所有交易 -> 可转走所有资产
```

更合理的方向是：

```text
智能账户 / 多签 / 限额 / 白名单 / Session Key
```

目标不是让 Agent “更自由”，而是让 Agent “被限制在可审计、可回滚、可限额的范围内”。

## 最小判断规则

判断一个账户模型是否适合交给自动化或 Agent 辅助时，至少看：

- 是否有单点私钥风险。
- 是否能设置额度限制。
- 是否能限制目标合约或目标地址。
- 是否能限制可调用函数。
- 是否能暂停、撤销或恢复权限。
- 是否能审计执行记录。
- 是否需要多人批准。

## 常见误区

### 误区 1：多签就绝对安全

多签降低的是单个 key 的风险，不是消除所有风险。

仍然可能出问题：

- owner 都由同一个人或同一台机器控制。
- 阈值设置太低。
- 多签模块被恶意授权。
- owner 被钓鱼诱导签名。

### 误区 2：智能账户一定比 EOA 安全

智能账户可以更安全，也可以更复杂。

安全来自规则设计和实现质量，不是来自“智能账户”这个名字。

### 误区 3：Session Key 就等于可以放心自动化

Session Key 只有在限制清楚时才有意义：

- 能做什么。
- 不能做什么。
- 额度多少。
- 有效期多久。
- 能否撤销。

### 1. EOA、智能账户、多签分别由什么控制

学员回答：

- EOA 由单个私钥控制。
- 智能账户由合约逻辑控制。
- 多签由多个私钥控制。

评价：

- 通过。
- 更精确地说，多签由多个 owner 加阈值规则共同控制，不只是“多个私钥”本身。

### 2. 为什么 EOA 是单点私钥风险

学员回答：

- 因为只要一个私钥泄露了，就完全失去了权限安全。

评价：

- 通过。
- EOA 的核心问题是控制权集中在一把 key 上，缺少原生限额、白名单、多人审批和可恢复机制。

### 3. 多签降低了什么风险，没有消除什么风险

学员回答：

- 降低了单点攻击风险，但是不能防范三个 owner 都在一台电脑或者都由一个人控制。

评价：

- 通过。
- 多签还需要关注 owner 分布、阈值设置、模块权限、owner 被钓鱼诱导签名等风险。

### 4. 智能账户相比 EOA 多了哪些可编程权限能力

学员回答：

- 每日限额。
- 白名单。
- Session Key 失效时间。

评价：

- 通过。
- 这些能力都属于智能账户或账户抽象方向常见的权限边界。

### 5. Session Key 安全使用需要限制什么

学员回答：

- 能做 / 不能做什么。
- 额度多少。
- 有效期多少。
- 是否能撤销。

评价：

- 通过。
- 如果用于真实系统，还可以补充目标合约、目标地址、可调用函数和调用频率限制。

### 6. 场景判断

学员回答：

```text
a. 个人用测试网练习转账：EOA
b. 团队管理项目 treasury：多签
c. 给 AI 助手一个每天最多 0.01 ETH 的操作权限：智能合约
d. 临时参加一个小额测试网活动：EOA
```

评价：

- 通过。
- 第 c 项更准确的表达是“智能账户”或“带限额规则的智能账户”，不是泛指任意智能合约。

### 7. 为什么 AI x Web3 助手不应该直接控制高权限 EOA

学员回答：

- 如果合约有 bug、升级权限混乱、模块被恶意授权，也可能更危险。给高权限 key 不可控。

评价：

- 通过，但需要修正表达重心。
- 本题核心不是智能合约本身风险，而是高权限 EOA 一旦被 Agent、脚本或运行环境误用，就缺少限额、白名单、撤销、多人审批和可审计边界。
- 更准确的一句话：AI x Web3 助手不应直接控制高权限 EOA，因为 EOA 的一把私钥代表几乎全部控制权，误操作或泄露后没有内建权限限制。
