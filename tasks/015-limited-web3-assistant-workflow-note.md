# 015｜受限 Web3 助手或小 workflow 设计

日期：2026-05-27

状态：学习中。

## 本节目标

本节只解决一个问题：

```text
如何设计一个“受限”的 Web3 助手或小 workflow。
```

这里的重点不是让助手能力最大，而是让它的权限边界清楚：

- 它可以做什么。
- 它不能做什么。
- 哪些地方必须人工确认。
- 它如何记录日志和 proof。
- 出错时如何停止。

## 主线定义

一个受限 Web3 助手，不是“自动替我操作钱包”的助手。

更合理的定义是：

```text
它可以解释、准备、检查、记录；
但不能直接签名、授权、转账、部署或替用户点击钱包确认。
```

一个最小 workflow 可以是：

```text
读取用户目标 -> 生成操作草稿 -> 生成检查清单 -> 等待人工确认 -> 记录结果 -> 整理 proof
```

## 设计一个助手时先问的 5 个问题

### 1. 目标是什么

目标要具体。

好例子：

```text
帮我整理一笔 Sepolia 测试网交易 proof。
```

差例子：

```text
帮我操作钱包。
```

目标越模糊，权限越容易失控。

### 2. 输入是什么

输入应该尽量是可公开或低风险材料。

可以输入：

- 交易哈希。
- 区块浏览器链接。
- 合约地址。
- ABI。
- 用户自己描述的目标。
- 截图中不含敏感信息的公开字段。

不应输入：

- 私钥。
- 助记词。
- API Key。
- 钱包备份文件。
- 未打码身份信息。

### 3. 输出是什么

输出应该是草稿、检查清单、解释或 proof。

例子：

- 操作步骤草稿。
- 风险检查清单。
- 合约函数解释。
- proof Markdown。
- 失败原因复盘。

输出不应该直接是：

- 已签名交易。
- 自动授权结果。
- 自动转账结果。
- 自动部署结果。

### 4. 工具权限是什么

工具权限要尽量只读。

低风险工具：

- 读取区块浏览器。
- 查询交易状态。
- 解码 ABI / calldata。
- 生成 Markdown。
- 检查公开链接。

高风险工具：

- 发送交易。
- 签名消息。
- 授权 Token。
- 管理私钥。
- 点击钱包确认。

### 5. 停止条件是什么

必须提前定义什么时候停。

例子：

- 网络不一致时停止。
- 合约地址不明确时停止。
- 参数无法解释时停止。
- `value` 非预期不为 0 时停止。
- 出现授权或转账高风险时停止。
- 区块浏览器显示 Failed 时停止。

## 最小规格模板

```markdown
# 受限 Web3 助手规格

## 目标

## 用户输入

## 允许动作

## 禁止动作

## 人工确认点

## 风险检查清单

## 日志与 proof

## 停止条件
```

## 示例：测试网交易 proof 助手

目标：

```text
根据用户提供的测试网 transaction hash，整理可提交的 proof。
```

允许动作：

- 查询交易哈希。
- 读取区块浏览器字段。
- 判断 Status。
- 整理 Network、From、To、Value、Gas、Block。
- 生成 Markdown proof。

禁止动作：

- 不创建钱包。
- 不保存私钥。
- 不签名。
- 不发送交易。
- 不授权 Token。
- 不替用户提交 WCB。

人工确认点：

- 用户确认交易确实属于自己的测试操作。
- 用户确认 proof 内容可以公开。
- 如果要提交到 WCB，先预览内容并取得确认。

停止条件：

- hash 查不到。
- 网络不明确。
- Status 不是 Success。
- 截图或输入中包含敏感信息。
- 用户要求助手直接签名或转账。

## 常见错误

### 错误 1：助手目标写得太大

不好的目标：

```text
帮我管理 Web3 钱包。
```

问题是权限范围不清楚。

更好的目标：

```text
帮我检查一笔测试网交易是否成功，并整理 proof。
```

### 错误 2：只写允许动作，不写禁止动作

禁止动作比允许动作更重要。

因为 Web3 的风险通常来自越权：

- 从解释变成签名。
- 从草稿变成自动提交。
- 从读取变成写入。
- 从小额测试变成主网操作。

### 错误 3：没有停止条件

没有停止条件的 workflow 很容易在异常状态下继续执行。

正确方式：

```text
不确定 -> 停
网络不对 -> 停
参数看不懂 -> 停
权限过高 -> 停
```

## 本节产出

本节最终产出应该是一份小型 workflow 规格，而不是一个复杂产品。

最小合格版本包括：

- 目标。
- 输入。
- 允许动作。
- 禁止动作。
- 人工确认点。
- 停止条件。
- proof 输出格式。
