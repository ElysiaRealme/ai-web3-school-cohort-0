# 016｜AI 可交互学习产物：页面、CLI、流程图、Quiz 或概念卡片

日期：2026-05-27

状态：已完成。

## 本节目标

本节只解决一个问题：

```text
如何把 Week 1 的一个知识点做成一个“可交互学习产物”。
```

这里的“可交互”不要求复杂。它可以是：

- 一个小网页。
- 一个 CLI 小问答。
- 一个 Mermaid 流程图。
- 一个 Quiz。
- 一个概念卡片选择器。

重点不是炫技，而是让别人可以通过操作、选择、回答或查看反馈来学习一个概念。

## 产物应该解决什么问题

普通笔记是：

```text
我写给自己看。
```

可交互学习产物是：

```text
别人可以通过它练习、判断、复习或验证理解。
```

例子：

```text
用户选择一个操作：签名 / 授权 / 交易
页面显示：风险等级、是否需要人工确认、应该检查哪些字段
```

这就比单纯写一段解释更有学习效果。

## 最小产物标准

一个合格的最小产物至少包含：

- 一个明确主题。
- 一个可操作入口。
- 一个反馈结果。
- 一个安全边界说明。
- 一个 README 或说明笔记。
- 记录哪些内容由 AI 生成，哪些由学员修改。

## 主题选择

建议从 Week 1 已学内容里选一个小主题。

可选主题：

1. Signature / Approval / Transaction 风险判断 Quiz。
2. Web3 操作安全检查清单生成器。
3. 测试网交易 proof 字段检查器。
4. AI x Web3 最小流程图解释器。
5. EOA / 智能账户 / 多签场景判断题。
6. 受限 Web3 助手规格生成器。

不要选太大的主题。

不推荐：

```text
做一个完整 Web3 钱包。
```

推荐：

```text
做一个测试网交易 proof 字段检查器。
```

## 交互形式对比

| 形式 | 优点 | 风险 / 成本 | 适合本节吗 |
|---|---|---|---|
| Markdown + Mermaid | 快，容易提交 | 交互弱 | 可作为最低限度 |
| CLI Quiz | 简单，可运行 | 需要本地命令 | 适合 |
| 静态 HTML 页面 | 直观，可公开 | 需要写一点前端 | 很适合 |
| React App | 扩展性好 | 对本节偏重 | 不优先 |
| 合约 / 链上 DApp | 真实但复杂 | 容易超纲 | 不建议本节做 |

## 推荐最小方案

本节建议选择：

```text
测试网交易 proof 字段检查器
```

原因：

- 你已经完成了真实 Sepolia 测试交易。
- 有现成 proof 字段。
- 风险低，不需要链上写入。
- 可以做成简单 HTML 或 Markdown checklist。
- 能直接服务 WCB proof。

最小交互逻辑：

```text
用户勾选是否包含 Network
用户勾选是否包含 Transaction Hash
用户勾选是否包含 Explorer Link
用户勾选是否包含 Status
用户勾选是否没有敏感信息
系统给出：可提交 / 需要补充
```

## 小网页结构示例

只看结构，不要求现在直接写完整代码：

```html
<h1>测试网交易 Proof 检查器</h1>

<label>
  <input type="checkbox" />
  包含 Network
</label>

<label>
  <input type="checkbox" />
  包含 Transaction Hash
</label>

<button>检查</button>

<p>这里显示检查结果</p>
```

这个例子的重点：

- `checkbox` 代表用户输入。
- `button` 触发检查。
- `p` 显示反馈。

## 伪代码

```text
如果所有必填项都勾选
    显示：可以作为 proof 草稿
否则
    显示：还缺哪些字段
```

如果加入安全边界：

```text
如果勾选“包含助记词 / 私钥 / API Key”
    显示：不能提交，请移除敏感信息
```

## AI 参与说明

WCB 任务通常会关注 AI 参与和人工复核。

需要记录：

- AI 生成了哪些内容：结构、文案、代码草稿、检查项。
- 学员修改了哪些内容：字段、规则、样式、边界说明。
- 哪些输出不可靠：AI 可能漏掉风险字段或误判 proof 完整性。
- 下一步如何改进：增加更多题目、增加真实 hash 校验、增加多网络支持。
