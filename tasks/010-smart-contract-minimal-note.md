## 第一轮检查题回答

### 1. 智能合约和普通后端服务最大的区别是什么？

学员回答：

- 状态公开、执行公开、不可篡改。

补充：

- 智能合约部署在链上，代码、状态变化和交易调用通常都可以被公开验证。
- 普通后端服务通常由服务方控制数据库、日志、部署和权限；智能合约的状态变更则通过链上交易完成，并由网络共识记录。
- “不可篡改”需要精确理解：已上链历史不可随意改写，但部分合约可能有升级代理、管理员权限或暂停机制，因此仍需检查合约权限和升级能力。

### 2. 读取合约和写入合约的区别是什么？

学员回答：

- 读取合约是查看链上数据，并不改变区块链数据；写入合约是发起一笔交易，会改变链上数据。

补充：

- 读取合约通常是 `call`，例如读取余额、owner、配置参数、allowance，不改变链上状态，通常不需要 Gas 或钱包确认。
- 写入合约通常是 transaction，例如 mint、transfer、approve、setValue、deposit，会改变链上状态，需要钱包签名、网络执行和 Gas。

### 3. 为什么部署合约也是一笔 transaction？

学员回答：

- 因为部署合约是创建了一个新的合约账户，也会有一次完整的状态变更，需要支付 Gas fee。

补充：

- 部署合约会把合约代码写入链上状态，并生成新的合约地址。
- 合约部署不是“上传文件”，而是一次链上状态变更，因此需要交易、Gas、交易哈希和区块浏览器验证。

### 4. Agent 生成合约调用脚本后，钱包确认前要检查哪些字段？

学员回答：

- `chainId`：是不是正确网络，例如 Ethereum mainnet、Arbitrum、Base、Sepolia。
- `from`：是不是自己的预期钱包。
- `to`：是不是预期合约地址。
- `value`：是否会转出原生币，例如 ETH、BNB、MATIC。
- `data`：是否能解码成预期函数。
- `gas 设置`：是否异常高。
- `nonce`：是否会覆盖或卡住已有交易。

补充：

- 对普通合约调用，`to` 应是预期合约地址；对合约部署交易，`to` 通常为空或使用合约创建语义，需要检查将要部署的 bytecode 和 constructor 参数。
- 如果 `data` 解码出 `approve`，还需要额外检查 spender、Token、授权额度和是否无限授权。
- 如果 `value` 不为 0，需要明确是否确实要转出原生币。
- `nonce` 通常由钱包自动管理，但在自定义脚本、批量交易或替换交易时需要特别谨慎。

## 人工确认清单

在执行任何合约写入、部署或高权限调用前，至少检查：

- 目标网络：`chainId` 是否符合预期。
- 发起账户：`from` 是否是预期钱包。
- 目标地址：`to` 是否是预期合约；部署交易则检查 bytecode 和 constructor。
- 原生币转出：`value` 是否符合预期。
- 调用数据：`data` 是否能用 ABI 解码成预期函数和参数。
- 授权风险：如果是 `approve`，检查 spender、Token、额度、有效期或无限授权。
- Gas：Gas limit / max fee 是否异常。
- Nonce：是否会替换、覆盖或卡住已有交易。
- 权限影响：本次调用是否会改变 owner、admin、allowance、资产余额或关键配置。
- 验证材料：是否能在区块浏览器中查看 transaction hash、状态、Gas、from / to、事件和状态变化。

## AI x Web3 边界

Agent 可以做：

- 解释合约 ABI。
- 生成合约读取或写入脚本草稿。
- 解码 calldata。
- 提醒检查网络、地址、函数、参数、value、Gas、nonce 和授权风险。
- 根据 transaction hash 整理 proof。

Agent 不应自动做：

- 部署合约。
- 写入合约。
- 签名交易。
- 授权 Token。
- 转账。
- 替用户点击钱包确认。
