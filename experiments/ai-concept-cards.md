# 008｜AI 基础概念卡片

日期：2026-05-23

## 目标

将 Week 1 中与 AI 相关的基础概念整理成可复习、可扩展、可用于 WCB proof 的概念卡片。学员是 AI Based，因此本文件不展开入门解释，而强调和 AI x Web3 workflow 的连接点。

## 卡片 1：LLM

定义：

- LLM 是基于上下文进行 token 预测和生成的大语言模型。它可以理解、总结、生成文本和代码，也可以辅助推理，但不等于事实数据库或确定性执行器。

为什么重要：

- AI x Web3 应用中，LLM 常作为解释层、规划层、代码生成层或交互层。

常见误区：

- 把 LLM 的自信回答当成事实。
- 让 LLM 直接决定高权限链上动作。

AI x Web3 连接点：

- LLM 可以解释交易、ABI、合约调用和区块浏览器记录，但最终状态必须由链上数据验证。

## 卡片 2：Prompt

定义：

- Prompt 是当前任务指令，用来告诉模型目标、约束、输入材料和输出格式。

为什么重要：

- Prompt 决定模型当前要解决什么问题，以及是否保留安全边界。

常见误区：

- 只写“帮我做”，没有明确来源、输出格式、验证要求和禁止动作。

AI x Web3 连接点：

- 涉及钱包、签名、授权、转账和合约写入的 prompt 必须显式声明：AI 只能生成草稿和解释，不能代替用户确认。

## 卡片 3：Context

定义：

- Context 是模型当前可见的工作材料，包括系统指令、用户输入、历史对话、文件内容、网页摘录和工具返回结果。

为什么重要：

- 模型不会天然知道所有事实；它只能基于当前 context 和已有能力生成答案。

常见误区：

- 把 secret、私钥、助记词、API key、未公开会议链接放进 context。

AI x Web3 连接点：

- Learning Agent 的 context 应包含课程来源、任务状态、公开 proof、学习目标和安全边界；不应包含任何钱包根秘密。

## 卡片 4：RAG

定义：

- RAG 是检索增强生成：先从外部资料检索相关内容，再把检索结果放进 context 让模型生成答案。

为什么重要：

- 对课程资料、Handbook、WCB 状态、代码库和链上资料进行问答时，RAG 能降低凭空编造的概率。

常见误区：

- 以为用了 RAG 就一定正确。检索结果可能过期、片面、无关或被错误引用。

AI x Web3 连接点：

- 查询合约文档、交易记录、WCB 任务和 Handbook 时，RAG 可以帮助定位材料，但 proof 仍需回到原始链接、commit 或区块浏览器记录。

## 卡片 5：Workflow

定义：

- Workflow 是人预先设计好的固定或半固定流程，模型只是流程中的一个节点。

为什么重要：

- 学习路线、WCB 查询、笔记生成、proof 打包和 GitHub 提交都适合 workflow 化。

常见误区：

- 把所有自动化都称为 Agent，忽略流程是否其实由人写死。

AI x Web3 连接点：

- 安全链路应是 workflow：AI 生成 -> 人工复核 -> 钱包确认 -> 链上执行 -> 区块浏览器验证。

## 卡片 6：Agent

定义：

- Agent 是能根据目标和中间状态动态规划、调用工具、读取结果并决定下一步的系统。

为什么重要：

- Agent 适合处理多来源读取、长流程任务拆解、状态维护和跨工具协作。

常见误区：

- 认为 Agent 越自主越好。越接近资产、权限和公开提交，越需要人工门控。

AI x Web3 连接点：

- Learning Agent 可以自主读取课程、整理笔记、打包 proof；但不能自主签名、授权、转账或写入合约。

## 卡片 7：Tool Use / Tool Calling

定义：

- Tool Use 是模型通过结构化调用外部工具来读取信息或执行动作，例如 API 请求、文件读写、浏览器访问、Git 操作。

为什么重要：

- 模型从“只回答”变成“能做事”，关键就在工具调用。

常见误区：

- 只关注模型答案，不审计工具调用参数和结果。

AI x Web3 连接点：

- 查询 WCB、读取 Notion、检查 Git 状态和生成 proof 可以使用工具；任何写入型平台操作或链上高权限动作都必须先预览并确认。

## 卡片 8：AI Coding

定义：

- AI Coding 是用模型辅助阅读代码、生成代码、修改文件、运行测试和解释报错。

为什么重要：

- 它能显著提高原型速度，尤其适合 Hackathon demo、脚手架、测试脚本和文档整理。

常见误区：

- 把生成代码等同于可靠代码，跳过审查、测试和真实路径验证。

AI x Web3 连接点：

- AI 可以生成合约交互脚本或前端连接钱包代码，但部署、授权、转账、合约写入前必须人工审查目标网络、合约地址、函数、参数和资产影响。

## 卡片 9：Evaluation

定义：

- Evaluation 是评估 AI 输出是否满足目标、是否可靠、是否可复现、是否越界的过程。

为什么重要：

- AI x Web3 的风险不只在“答错”，还在“错着执行”。因此评估必须覆盖事实、代码、权限、工具调用和 proof。

常见误区：

- 只看输出是否像样，不看来源、执行日志、测试结果和验证材料。

AI x Web3 连接点：

- 可靠的 evaluation 应至少检查：来源链接、工具调用结果、GitHub commit、WCB 状态、交易 hash、区块浏览器状态和人工确认记录。

## 总结

本组概念卡片服务于两个 Week 1 任务：

- `AI 向任务｜整理 AI 基础概念卡片`
- `AI x Web3 综合任务｜画出 AI x Web3 最小交叉流程图`

核心结论：

- AI 可以参与读取、解释、规划、生成和 proof 打包。
- Web3 权限动作必须保留人工确认。
- 所有可提交学习成果都应能回到公开链接、GitHub commit、WCB 状态或链上验证记录。
